# Domain Model

## Entity Hierarchy

```
User
│
├─ OrganizationMembership ──► Organization
│                                  │
│                                  ├─ Project
│                                  │    └─ ProjectSection
│                                  │         └─ Bug
│                                  │              ├─ BugComment
│                                  │              ├─ BugAttachment
│                                  │              └─ BugHistoryEntry
│                                  │
│                                  ├─ OrganizationInvitation
│                                  ├─ OrganizationAuditLog
│                                  └─ BugDataJob (import/export)
│
├─ Session
├─ AuthActionToken (email verify, password reset)
└─ ApiKey / AuthJwtConfig (settings)
```

## Key Concepts

### Organization
The top-level tenant boundary. All bugs, projects, and users exist within an organization. A user can belong to multiple organizations and switches between them via the active organization session.

### Project & Section
Bugs are always assigned to a Project and a ProjectSection (a sub-grouping within a project). Projects can be archived. Sections have a `displayOrder` for sorting.

### Bug
The core entity. Has a status lifecycle (New → Closed), severity, optional assignee, reporter, and belongs to a Project/Section. Supports comments, attachments, and a full history timeline.

### BugHistoryEntry
Immutable record of every change to a bug. Records `eventType` (prefixed `bug_` for user actions, `bug_import_` for bulk imports) and a list of changed fields with old/new values.

### TestingSession
An active browsing session by the Chrome extension. Evidence (screenshots, log snippets) captured during testing is linked to a bug as attachments via `TestingSessionLog`.

### OperationalAlert
Generated by the monitoring system when error groups exceed thresholds. Can be promoted to a Bug by an admin.

### BugDataJob
Async job record for bulk import or export operations. Tracks status (`queued → running → completed/failed`), warnings, errors, and a summary of counts.

### Notification Event / Delivery
When a bug changes state, a `NotificationEvent` is created and dispatched to eligible recipients. Each recipient gets a `NotificationDelivery` record tracking delivery status and retry history.

## Agent Concepts

### Claimable Bug
A bug that is available for an AI agent to work on. Agents claim bugs, update their `state` and `statusNote`, and release them.

### Agent State
Tracked via `updateBugAgentWork()`. Separate from the user-facing bug status — records the agent's current work phase.
